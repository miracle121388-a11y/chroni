# DueFlow 产品要求文档

## 1. 产品定位

DueFlow 是一个以桌宠为核心入口的本地 DDL 日程助手。用户把文件、图片、截图或文本拖给桌宠后，系统自动识别 DDL 并生成日程。

产品目标是方便快捷，不做复杂厚重的任务管理软件。

最终形态：

```text
桌宠负责交互入口
日程表负责轻量展示
控制中心负责必要设置和少量修正
```

最高原则：

- 桌宠必须极简，只负责接收输入、表达状态和给短反馈。
- 日程表必须极简，只展示用户马上需要看的日程信息。
- 控制中心也必须轻量，只保留必要设置和少量修正能力。
- 能由系统智能完成的事情，不让用户手动配置。
- 常用路径必须短，用户不应为了记录一个 DDL 进入复杂流程。

## 2. 核心闭环

```text
桌宠常驻桌面
  -> 用户拖入文件 / 图片 / 截图 / 文本
  -> 桌宠反馈收到并进入处理状态
  -> 系统自动解析内容
  -> LLM 自动抽取 DDL 和任务信息
  -> 系统直接生成或更新日程
  -> 日程表轻量展示最新结果
  -> 桌宠根据日程状态提醒用户
  -> 用户快速查看 / 完成 / 稍后提醒
  -> 需要少量修正或设置时进入控制中心
```

关键要求：

- 拖入内容后不设置人工确认步骤。
- 系统自动完成识别、抽取和生成日程。
- 识别失败、信息不足或模型不可用时，由桌宠给出明确短反馈。
- 日程表是轻量结果展示，不是复杂管理界面。
- 控制中心只承载必要设置和少量修正。

## 3. 桌宠要求

桌宠是产品主入口，但不展示复杂内容。

基础能力：

- 常驻桌面。
- 透明背景。
- 可拖动。
- 可隐藏。
- 可吸附屏幕边缘。
- 可接收拖拽输入。
- 根据日程状态切换表现。
- 右键打开控制中心。

桌宠不展示：

- 完整任务列表。
- 复杂日历。
- 任务详情。
- 历史记录。
- 模型或系统设置。

左键点击：

- 触发当前状态相关的小动作。
- 显示一句短气泡。
- 有临近 DDL 时提示最重要的一条。
- 无任务时给轻量反馈。
- 全部完成后给正反馈。

拖拽输入反馈：

```text
拖拽悬停
  -> 桌宠进入接收状态
释放内容
  -> 桌宠提示收到
解析中
  -> 桌宠进入处理状态
抽取成功
  -> 桌宠提示已加入日程
抽取失败
  -> 桌宠提示失败原因
```

失败反馈至少覆盖：

- 文件类型不支持。
- 文件无法读取。
- 图片 OCR 失败。
- 没有识别到明确 DDL。
- 关键信息不足。
- 模型或本地服务不可用。

## 4. 日程表要求

日程表只负责轻量展示，回答一个问题：

```text
我现在或最近需要注意什么？
```

日程表展示内容：

- 今天最重要的少量事项。
- 最近 3 到 7 天即将截止的少量关键事项。
- 已逾期事项的简洁提醒。
- 刚刚从拖拽输入生成的新事项。

日程表默认不展示全部任务。若内容超过轻量展示范围，应提供“更多”入口进入控制中心的简洁列表。

日程表允许的轻操作：

- 一键完成。
- 稍后提醒。
- 打开控制中心查看详情。

日程表不承担：

- 展开完整任务详情。
- 完整月历管理。
- 复杂周视图。
- 深度任务编辑。
- 批量管理。
- 来源记录管理。
- 模型、OCR、数据、诊断设置。

这些能力不进入日程表；只有必要部分进入控制中心。

### 4.1 推荐展示形式

日程表采用极简 DDL 清单。界面默认只展示 4 到 6 条事件，让用户几秒内看清当前最紧急、最重要的 DDL。

每个事件只显示：

- 事件名称。
- 重要性。
- 截止时间。
- 剩余时间。

默认排序规则：

```text
紧急程度 + 重要性
```

最需要关注的事件必须放在最上方。

支持的轻操作：

- 点击查看。
- 勾选完成。
- 快速添加。

如果事件超过默认展示数量，只显示最需要关注的 4 到 6 条，其余通过“更多”进入控制中心的简洁列表。

### 4.2 单条日程样式

单条日程使用紧凑清单行，不做复杂卡片：

```text
课程报告 PDF    重要    7月12日 23:59    剩余 2 天
```

字段要求：

- 事件名称：短标题，尽量控制在 12 到 16 个中文字符内。
- 重要性：高 / 中 / 低，或使用等价的简短标签。
- 截止时间：明确日期和时间。
- 剩余时间：已逾期、剩余 N 小时、剩余 N 天。

颜色只用于区分状态：

- 红色：24 小时内或已逾期。
- 橙色：3 天内。
- 灰色：普通。

不要使用大面积色块、复杂图表、完整月历或多层嵌套信息。

整体视觉采用小尺寸、圆角、半透明的桌面组件风格。

### 4.3 大模型介入边界

可以借助大模型做：

- 把长任务标题压缩成短标题。
- 结合紧急程度和重要性判断展示排序。
- 生成简短的事件名称。
- 判断事件重要性标签。
- 识别信息不足时生成短提示。

不让大模型做：

- 动态决定界面布局。
- 生成复杂长文解释。
- 替用户隐藏高风险或逾期事项。
- 直接修改日程规则。
- 改变固定字段结构。
- 输出不可预测的 UI 结构。

原则：

```text
大模型负责压缩信息
模板负责稳定展示
用户能从控制中心进行必要修正
```

## 5. 控制中心要求

控制中心不是复杂后台，只用于必要设置、少量修正和基础排错。用户只有在自动结果不符合预期，或需要调整偏好时才打开控制中心。

控制中心只保留三个模块：

- 轻量修正：简洁事件列表，修改事件名称、重要性和截止时间，删除误识别事件，对单条输入重新识别。
- 基础偏好：桌宠开关、桌宠形象入口、提醒、勿扰时间、快捷键。
- 服务状态：模型 / OCR 基础状态、本地数据和隐私说明、必要排错提示。

控制中心不做完整详情页，不要求用户长期停留使用。能自动判断和自动处理的内容，不放成设置项。

控制中心不应包含：

- 完整项目管理。
- 复杂日历系统。
- 周报 / 月报 / 统计看板。
- 批量任务管理。
- 高级插件市场。
- 大量模型参数。
- 面向开发者的复杂诊断面板。

原则：

```text
默认智能处理
必要时轻量修正
设置少而清晰
```

## 6. 平台展示形态

### 6.1 Windows

Windows 版本采用类似 QQ 的侧边抽屉形态。

要求：

- 日程表吸附在屏幕侧边。
- 默认收起成侧边抽屉。
- 鼠标移动到屏幕侧边时展开。
- 鼠标离开后自动收起。
- 展开内容就是轻量日程表。
- 抽屉内容必须简洁，不做完整管理后台。
- 桌宠与侧边抽屉协同工作。

Windows 重点体验：

```text
桌宠接收输入
  -> 自动生成日程
  -> 侧边抽屉日程表更新
  -> 用户移到侧边快速查看
```

### 6.2 macOS

macOS 版本采用桌面组件形态展示日程表。

要求：

- 日程表以桌面组件形式常驻展示。
- 桌面组件只展示今日任务、近期 DDL 和关键提醒。
- 桌面组件必须轻量，不做完整管理后台。
- 桌宠仍然负责输入、互动和提醒。
- 控制中心承载必要设置和少量修正。

macOS 重点体验：

```text
桌宠接收输入
  -> 自动生成日程
  -> 桌面组件同步更新
  -> 用户直接在桌面快速查看
```

## 7. 桌宠形象与交互设计

桌宠形象暂未确定，后续会调研可合法使用的形象、素材和展示方案。当前阶段只固定交互状态，不绑定具体实现方式。

可能的表现方式：

- 静态图片切换。
- GIF / APNG。
- PNG 序列帧。
- Live2D。
- Spine。
- 自定义动画组件。

需要预留的交互状态：

- `idle`：待机。
- `clicked`：左键互动。
- `hover_accept`：拖拽悬停。
- `processing`：处理中。
- `success`：处理成功。
- `confused`：识别失败或信息不足。
- `deadline_near`：DDL 临近。
- `overdue`：逾期。
- `celebrating`：任务完成。
- `sleeping`：勿扰或休息。

后续确定形象后，只需要将这些状态映射到具体动作或素材。

## 8. 数据处理原则

输入内容统一进入自动处理流水线：

```text
输入内容
  -> 文本解析 / OCR
  -> 去重
  -> LLM 抽取
  -> 任务生成
  -> 日程生成
  -> 紧急程度和重要性排序
  -> 日程表更新
  -> 桌宠状态更新
```

由于不再设置人工确认步骤，系统需要增强事后修正能力：

- 用户可以在日程表中快速完成或稍后提醒。
- 用户可以从日程表跳转到控制中心做必要修正。
- 用户可以在控制中心删除误识别任务。
- 用户可以在控制中心修改事件名称、重要性和截止时间。
- 用户可以在控制中心查看原始输入摘要。
- 用户可以在控制中心重新识别某条输入。

## 9. 当前开发方向

后续开发应围绕以下目标收敛：

1. 桌宠成为默认启动后的主入口。
2. 拖拽输入后自动生成日程。
3. 桌宠和日程表保持极简。
4. Windows 实现侧边抽屉式轻量日程表。
5. macOS 实现桌面组件式轻量日程表。
6. 控制中心只保留必要设置、少量修正和基础排错。
7. 桌宠形象系统先保留状态接口，等形象方案确定后再接入具体展示。
